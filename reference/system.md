# Agent 框架 —— 论文 Method 部分依据

> 覆盖范围：`core/`（认知组件）+ `memory/`（记忆系统）。用于引导其他 AI 撰写论文第三部分 Method 中 Agent 框架的内容。
> 本文档只描述各组件的定位、职责和设计逻辑，不涉及数据流格式、运行模式切换、代码实现细节。

---

## 一、整体构成

Agent 框架由 **5 个认知组件** 和 **2 类记忆库** 组成。认知组件编排为一个有向无环流水线，具备审核失败时的自动回退修正能力（最多 3 次重试）。记忆库在规划阶段以检索增强生成（RAG）的方式注入历史经验。

**认知组件及其定位**：

| 组件 | 文件 | 定位 |
|------|------|------|
| 视觉分析 | `core/vision.py` | 感知 —— 从原始图像到结构化火场态势 |
| 任务规划 | `core/planning.py` | 决策 —— 从态势理解到多机协同任务方案 |
| LLM 语义审核 | `core/validation.py` | 反思 —— 从战术合理性角度审查方案质量 |
| 确定性规则校验 | `core/validation.py` | 验证 —— 以规则引擎进行结构化合法性检查 |
| 运行时重规划 | `core/replanning.py` | 自适应 —— 响应突发故障，局部调整任务分配 |

**记忆库及其定位**：

| 记忆库 | 目录 | 定位 |
|--------|------|------|
| 情景记忆（Episodic Memory） | `memory/planning_memory/` + `memory/replanning_memory/` | 存储历史成功案例（输入-输出对），按场景或故障特征检索 |
| 经验记忆（Lesson Memory） | `memory/lesson_memory/` | 存储从每次运行中抽象出的跨场景通用经验教训，始终写入 |

流水线顺序为：视觉分析 → 任务规划 → LLM 审核 → 规则校验。任一审核节点判定失败时，反馈作为上下文注入任务规划节点触发重试。重规划为独立模块，在仿真执行阶段由外部事件触发，不接入上述默认流水线。两类记忆库分别在任务规划和重规划节点中提供 RAG 经验检索。

## 二、认知组件

### 2.1 视觉分析（`vision.py`）

**定位**：感知层。将非结构化的视觉输入（航拍图）转化为结构化的火场态势描述。

**做了什么**：调用多模态 LLM 同时完成两项任务——(1) 检测图像中所有火点并输出归一化坐标；(2) 根据火点空间分布密度进行聚类，将连续火场划分为 2~5 个离散任务区域。每个区域标注风险等级（High 高风险攻坚区 / Low 低风险控制区 / Monitor 无火监测区）及所含火点坐标列表。

**替代路径**：当侦察阶段已通过 YOLO 完成火点检测时，视觉分析可跳过检测环节，仅由 LLM 基于已知火点坐标完成区域划分。

**为什么是 LLM 而非纯视觉模型**：区域划分不仅依赖火点位置，还需理解烟雾扩散范围、植被分布、地形边界等语义信息，需要视觉-语言联合推理能力。

### 2.2 任务规划（`planning.py`）

**定位**：核心决策层。将火场态势转化为可执行的多无人机协同任务方案。

**做了什么**：调用 LLM 为异构无人机集群中的每架飞机生成完整的任务链。LLM 被约束在一个预定义的动作函数白名单内（如 `takeoff`、`fly_to_point`、`hover_and_wait`、`drop_water_bomb`、`SearchArea`、`return_to_base`、`land`、`resupply`），确保生成的行为始终在可控语义空间内。生成内容涵盖任务分配（谁做什么）、时序编排（什么顺序做）和续航管理（何时返航补给）。

**记忆增强**：规划时从情景记忆中检索相似火场场景的历史成功方案作为战术参考，从经验记忆中检索跨场景通用经验作为规划偏好。两种记忆均以自然语言形式注入 LLM 的 prompt 上下文。

**失败修正**：当下游审核判定失败时，审核反馈作为对话历史的一部分回传给 LLM，引导其进行定向修正而非从头随机重试。

**为什么用代码而非结构化 JSON 输出任务方案**：无人机任务链天然是过程式的——起飞、飞往 A、悬停、投弹、飞往 B、搜索区域 C、返航——用 Python 函数调用序列表达比嵌套 JSON 更紧凑且可被下游模拟器直接执行（通过沙箱 exec），减少了一层解析转换。

### 2.3 LLM 语义审核（`validation.py` 中的 `llm_validation`）

**定位**：反思层。以独立 LLM 实例从战术合理性角度审查规划输出，是审核-修正闭环的第一道关口。

**审核维度**（5 项）：
1. 区域覆盖完整性 —— 所有含火点的区域是否均被分配任务
2. 资源-任务匹配 —— 灭火型/侦察型无人机是否各司其职
3. 弹药覆盖充分性 —— 灭火弹总容量是否覆盖火点数量，投弹后是否有复查搜索
4. 侦察合理性 —— 侦察机搜索分配是否合理，是否存在无效重复搜索
5. 整体策略健康度 —— 是否存在资源浪费、兵力不足或无人机类型错配

输出 PASS/FAIL 判定及具体问题说明。FAIL 时反馈回任务规划节点触发修正。

**为什么需要独立 LLM 而非让规划 LLM 自审**：同一 LLM 实例倾向于确认自己的输出是合理的（self-consistency bias）。独立实例 + 独立的审核 prompt 构造了一个对抗性审查视角，更可能发现规划盲区。

### 2.4 确定性规则校验（`validation.py` 中的 `rule_validation`）

**定位**：验证层。以非 LLM 的确定性规则引擎进行结构化合法性检查，是审核-修正闭环的第二道关口。

**校验项**（6 项）：
1. Python 语法编译 —— 确保生成代码可被解析
2. API 白名单 —— 确保未调用未定义的函数
3. 无人机 ID 合法性 —— 确保引用的无人机在机队中存在
4. 坐标范围 —— 确保所有坐标在 [0, 1] 归一化区间内
5. 任务链结构完整性 —— 确保以 takeoff 开头、land 结尾，投弹动作前有悬停确认
6. 搜索负载均衡 —— 有空闲无人机时，不允许单机承担多个搜索区域

**与 LLM 审核的互补关系**：LLM 审核回答"方案是否合理"（战术层面，可能漏判或误判），规则校验回答"方案是否合法"（结构层面，零漏判）。规则引擎不依赖 LLM，消除了幻觉导致的漏检风险。

### 2.5 运行时重规划（`replanning.py`）

**定位**：自适应层。在任务执行阶段响应突发情况，对任务方案进行局部修补而非推倒重来。

**触发条件**：由外部事件驱动（如仿真中某架无人机故障、电池耗尽、发现新增火点），通过自然语言描述注入。

**核心原则**：最小范围修改 —— 已执行任务视为不可撤销的历史事实；仅重新分配受影响无人机的待执行任务队列；未受影响的无人机完全不受扰动。

**记忆增强**：从情景记忆中检索相似故障场景的历史修复策略作为参考。

**为什么独立于主流水线**：重规划的输入不是原始图像和 zones，而是运行时实时状态快照（各无人机当前位置、电量、载荷、已完成/未完成任务）；其目标不是生成全新方案而是局部修补。两者在输入类型和任务性质上根本不同。

## 三、记忆系统

### 3.1 总体设计

记忆系统分为两类，性质不同：

- **情景记忆（Episodic Memory）**：存储具体的历史案例，即"输入 → 输出"对。分为初始规划案例和重规划修复案例两个物理库，但共享相同的存储机制和写入策略——仅任务成功时写入，保证参考样本的可靠性。
- **经验记忆（Lesson Memory）**：存储从每次运行中经 LLM 抽象提炼的通用经验教训，是跨场景的规划偏好而非具体案例回放。无论任务成败都写入，最大化学习机会。

每类记忆采用双层存储：JSONL 文件作为真实数据源（append-only），FAISS 向量索引作为可选加速检索层。索引构建失败不影响 JSONL 存储，系统自动退化为无 RAG 模式。

检索流程：将查询内容构造为自然语言文本 → 嵌入模型向量化 → FAISS 相似度搜索 Top-K → 按 ID 从 JSONL 回填完整记录 → 格式化为自然语言注入 LLM prompt。嵌入模型使用 Google Generative AI Embeddings（`gemini-embedding-001`）。

### 3.2 情景记忆

**统一语义**：存储"在某种情境下，什么样的方案被验证是有效的"。按情境类型分两个物理库，使用相同的存储机制和写入策略。

**初始规划库（Planning Memory）**：
- 检索依据：当前 zones（区域数量、风险分布、火点密度）+ 可用无人机配置
- 存储内容：zones、UAV 资源清单、生成的 mission_code、LLM 审核和规则校验的反馈
- 服务对象：任务规划节点

**重规划修复库（Replanning Memory）**：
- 检索依据：当前突发事件描述 + 运行时状态摘要 + zones 摘要
- 存储内容：突发事件描述、触发历史序列、重规划前运行时状态快照、受影响无人机列表、pending_tasks 变更前后完整对比
- 服务对象：运行时重规划节点

**写入策略**：仅任务成功时写入。两个物理库分开的必要性在于检索维度不同——初始规划检索的关键是"火场长什么样"，重规划检索的关键是"出了什么故障 + 当时系统处于什么状态"，混在一起会降低各自检索精度。

### 3.3 经验记忆（Lesson Memory）

**语义**：存储"从每次运行中抽象出的通用经验教训"，是跨场景的规划偏好而非具体案例。

**与情景记忆的本质区别**：

| | 情景记忆 | 经验记忆 |
|------|----------|----------|
| 存储内容 | 具体案例（输入-输出对） | 抽象经验（自然语言短句） |
| 写入条件 | 仅任务成功 | 始终写入（含失败） |
| 检索依据 | 情境特征（场景/故障） | 场景摘要 |
| 注入方式 | 作为参考案例 | 作为规划偏好 |
| 信息来源 | 任务运行原始产物 | LLM 对全部产物的综合提炼 |

**LLM 总结流程**：任务结束后，调用 LLM 综合分析全部运行产物——zones、初始 mission_code、runtime_state 终态、评分指标（耗时/能耗/错误报告）、LLM 审核和规则校验反馈、任务执行日志、重规划报告——从中提炼 3-6 条可复用的简短经验。若 LLM 总结调用失败，系统基于基础运行指标自动生成兜底经验，确保失败任务的信息不丢失。

**为什么失败任务也必须写入经验记忆**：成功的规划方案存入情景记忆作为正面参考，但失败任务中暴露的问题（如某类火场分布下负载估算偏低、某类故障下重规划修正过激进）同样是宝贵信息。经验记忆不是"推荐这么做"，而是"注意这些情况"，因此可以接纳负面样本。

## 四、框架层面的核心设计

以下要点按"在论文 Method 部分的可展开程度"排序：

1. **审核-修正闭环**：规划 LLM → 独立 LLM 语义审核 → 确定性规则校验 → 失败反馈回规划 LLM。两级审核分别在战术合理性和结构合法性上把关，形成有质量保障的自我修正循环。

2. **情景与经验双轨记忆**：情景记忆存储具体案例（仅成功写入，作为正面参考），经验记忆存储抽象教训（始终写入，作为风险提示）。两者在存储内容、写入策略和检索方式上互补，而非冗余。

3. **确定性校验兜底**：在 LLM 审核之后增加一层不依赖 LLM 的规则引擎，消除模型幻觉带来的漏检风险——LLM 审核可能漏判，规则校验不可能漏判。

4. **重规划的局部性**：运行时故障只触发受影响无人机的局部任务调整，已执行部分不可撤销，未受影响部分不受扰动。区别于"遇到问题就全局重新规划"的朴素方案，降低了响应延迟和计算开销。

5. **动作空间约束**：规划 LLM 的输出被限制在预定义的动作函数白名单内，从机制上杜绝了 LLM 生成不可执行指令的可能性。

6. **多模型分工**：不同认知阶段使用不同 LLM——视觉理解用 Gemini（多模态能力）、规划与审核用 DeepSeek（推理能力）、嵌入用 Google Embedding，而非单一模型承担所有任务。
